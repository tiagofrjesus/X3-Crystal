1. PORQUÊ ESTE FICHEIRO SEPARADO
------------------------------------------------------------------------
  O pedido original era alterar o "PIECE" standard, mas descobrimos (por
  teste direto no X3: renomear "PIECE" para "ZPIECE" na lista faz os
  dados desaparecerem, com o MESMO .rpt) que o motor de impressão do X3
  resolve a ligação/pasta com base no CÓDIGO do relatório registado em
  AREPORT (dicionário), não só no ficheiro .rpt. Por isso:

    - Reports-TEB/PIECE.rpt fica como estava (standard, intocado).
    - Reports-TEB/TEB_PIECE.rpt é a cópia com o novo layout (retrato +
      info da empresa), a implantar sob um CÓDIGO DE RELATÓRIO NOVO
      ("TEB_PIECE") registado à parte no dicionário X3 — ver secção 3.

2. ALTERAÇÕES DE LAYOUT (idênticas às descritas antes para o PIECE)

  a) Orientação: PaperSize=A4, Orientation=Portrait, dissociado da
     impressora, ModifyUserPaperSize(16838,11906) (altura,largura twips).

  b) Todos os objetos foram reescalados horizontalmente (Left e Width)
     por um fator R = 11906/16838 ≈ 0.7071 — EXCETO os objetos de Texto

     Width original (o desenho original já tinha essas caixas no limite
     exato do texto; encolher a largura cortava-o, ex. "Estab." → "Esta").

  c) Nova tabela NATIVA "BPADDRESS" (alias BPADDRESS_FCY, QualifiedName
     "TEB.BPADDRESS"), LEFT OUTER JOIN a partir de FACILITY:
        {FACILITY.BPAADD_0} = {BPADDRESS_FCY.BPAADD_0}
        {FACILITY.FCY_0}    = {BPADDRESS_FCY.BPANUM_0}
     (mesmo padrão já usado em TEB_PAG.rpt.) Entrou como tabela NATIVA
     (não Command) — lição já registada em TEB_ITM_ETIQx60.txt: o print
     engine do X3 só remapeia ligações de tabelas nativas para o DSN do
     folder em runtime.

  d) Nova fórmula "lbl_companyinfo" (idêntica à de TEB_PAG.rpt):
        {COMPANY.CPYNAM_0} + " - " + {BPADDRESS_FCY.BPAADDLIG_0} + " - "
        + {BPADDRESS_FCY.POSCOD_0} + "-" + {BPADDRESS_FCY.CTY_0} + " - "
        + {COMPANY.NID_0}

  e) Novo campo "lblcompanyinfo1" no Page Footer (Left=105, Width=11700,

3. REGISTO NO DICIONÁRIO X3 (feito, exceto o passo 4 — ver abaixo)
------------------------------------------------------------------------
  AREPORT/AREPORTD/AREPORTV para "TEB_PIECE" já foram copiados de "PIECE"
  via COPRPT + GESARP (confirmado por SQL: AREPORT ok, AREPORTD com as

  9 GESGPC, já corrigido).

  CAUSA RAIZ DO "SEM DADOS" (confirmada por SQL + código-fonte):
  TEB.AREPORTM (tabela temporária de chave de impressão) tem chave
  NUMREQ+USR+RPTCOD+NUMLIG — cada relatório tem as SUAS PRÓPRIAS linhas.
  Consultando: RPTCOD_0='PIECE' tinha 64 linhas (a mais recente da
  própria sessão de teste); RPTCOD_0='TEB_PIECE' tinha ZERO, mesmo
  depois de várias impressões de teste via AIMP.

  A geração dessa linha é feita pelo subprograma TRTINI_0=TRTJOULEG
  (o "processo de inicialização" do relatório, chamado pelo template
  standard antes de invocar o Crystal Reports). Analisando o código-
  fonte (Reports-BaseX3/TRTJOULEG.src, fornecido pelo utilizador):

    Subprog IMPRIME(NBPAR,PARAMETRE)
    ...
    Case ETAT                          # ETAT = código do relatório (X3ETA)
     When "JOUGEN"
      ...
     When "LOT","PIECE"
      ...
      Gosub TRTJOU From RPTLEG         # É AQUI que se gera a linha AREPORTM
    Endcase
    End

  O Case só reconhece "JOUGEN", "LOT" e "PIECE" — SEM Default/Else. Para
  ETAT="TEB_PIECE" o Case não bate em nada, a rotina não faz nada, nunca
  chama TRTJOU, e por isso nunca há linha em AREPORTM → relatório mostra
  só o layout estático, sem dados. TRTJOULEG é standard — não pode ser
  alterado (regra: nunca modificar SUBxxx/processos standard; perde-se
  em updates). Não tem entry point registado em AENTREE, logo não há
  forma "limpa" (SPExxx) de o interceptar.

  FIX v1 (sem tocar em nada standard): criado Reports-TEB/ZTRTJOULEG.src —
  um subprograma NOVO (prefixo Z) que replica exatamente o ramo "PIECE"
  do TRTJOULEG original (mesma validação de empresa, mesmo
  "Gosub TRTJOU From RPTLEG"), adaptado para reconhecer ETAT="TEB_PIECE".

  SEGUNDO BLOQUEIO (só apareceu depois de resolver o primeiro — confirmado
  com ZTRTJOULEG a correr sem erro mas continuando com 0 linhas em
  AREPORTM). Pedido o código-fonte de RPTLEG à Sage (suporte #74969).
  RESPOSTA OFICIAL DA SAGE:

    "The report is empty because it is linked to the script TRTJOULEG.
    [...] Then the script RPTLEG is called: it also contains
    instructions depending on the report code. So, it is not so simple
    to duplicate this report. [...] Depending on what the customer wants
    to do, he can also customize the standard report, which is not often
    modified (last time, it was for 2021 R4)."

  E, depois de enviar o RPTLEG.src pedido:

    "A lista PIECE tem um tratamento standard associado que executa
    determinadas ações em função da lista a imprimir. Esse tratamento
    controla o nome exato da lista. [...] Se o que pretendem é apenas
    efetuar ajustes ao ficheiro .rpt, a nossa sugestão é que não
    dupliquem o registo no dicionário de listas. Mantenham o PIECE e
    alterem apenas o ficheiro .rpt associado."

  ==> Recomendação oficial da Sage: NÃO duplicar o código do relatório.
      Decisão do cliente: manter na mesma o código TEB_PIECE em separado
      (para poder imprimir via AIMP sem depender do botão da GESGPC/
      GESGAS), assumindo o esforço extra de contornar o 2º bloqueio.

  Analisando Reports-BaseX3/RPTLEG.src (fornecido pela Sage), a label
  $TRTJOU (para onde o nosso Gosub salta) tem o SEU PRÓPRIO Case ETAT:

    $TRTJOU
    ...
    Case ETAT
     When "LOT", "PIECE2" : CRITERE="1=1"     # sem restrição de diário
     When Default
      ... monta CRITERE a partir de "joudeb"/"joufin" ...
    Endcase
    ...
    Gosub TR2   # percorre [JOU] Where evalue(CRITERE) e escreve AREPORTM

  Para ETAT="TEB_PIECE" cai no Default; os parâmetros "joudeb"/"joufin"
  NÃO existem no dicionário deste relatório (nem no do PIECE original) →
  CRITERE não bate com nenhum diário real → Filtro [JOU] vazio → TR2
  nunca escreve nada em AREPORTM. Sem erro, porque é só um filtro vazio.

  FIX v2 (ainda sem tocar em nada standard): confirmámos que o
  TEB_PIECE.rpt NUNCA verifica {AREPORTM.RPTCOD_0} nos joins nem no
  record selection (só usa NUMREQ_0/CLEA1_0/CLEA3_0) — por isso é
  inofensivo a linha ficar gravada com RPTCOD="PIECE". ZTRTJOULEG.src
  agora força ETAT="PIECE" (variável Local NOSSA, dentro do NOSSO
  subprograma) só no instante imediatamente antes do
  "Gosub TRTJOU From RPTLEG", enganando SÓ o Case desse RPTLEG para
  entrar no ramo "1=1" — sem alterar uma linha de TRTJOULEG nem RPTLEG.
